
מובן. אתה צודק — צריך שהדשבורד ישקף את טבלת פייסבוק בדיוק כמו בעמוד הטבלה, בלי לוגיקה ש”ממציאה” סיווג אחר.

### מה הבעיה כרגע
1. יש שגיאת Build ב-`SharedTable.tsx` בגלל שימוש ב-`getCampaignType` שלא קיים.
2. ב-`DashboardView.tsx` יש עדיין לוגיקת סיווג סטטית (`facebook_insights => leads`) שעלולה לגרום לדשבורד להציג נתונים שונה מהטבלאות בפועל.

### תוכנית תיקון
1. **תיקון מיידי של שגיאת Build**
   - קובץ: `src/pages/SharedTable.tsx`
   - להסיר תלות ב-`getCampaignType` ולחשב תצוגת לידים/איקומרס מתוך הנתונים עצמם (אותה לוגיקה שכבר קיימת לזיהוי ecommerce לפי `campaign_type`, `purchases`, `purchase_value`, `add_to_cart`).
   - זה פותר את `TS2552` ומחזיר קומפילציה תקינה.

2. **יישור הדשבורד לטבלאות “כמו שהן”**
   - קובץ: `src/pages/DashboardView.tsx`
   - במקום לאחד את כל טבלאות הפלטפורמה למבנה אחד, אציג את הנתונים **לפי טבלה מקורית** (כל טבלה עם השדות שלה ובאותו סדר מה-`crm_fields`).
   - המשמעות: טאבת Facebook בדשבורד תציג את הרשומות בדיוק כמו בעמוד טבלת Facebook (אותם עמודות, אותו תוכן, אותו סדר לוגי), ולא תצוגה “ממוזגת” שעלולה לשנות תמונה.

3. **ביטול השפעת סיווג סטטי על Facebook**
   - לעדכן את חישובי הדשבורד כך שלא יתבססו על `getCampaignType` קשיח עבור Facebook.
   - סיווג (אם צריך לתצוגות סיכום) יתבצע רק מנתוני הרשומה בפועל, כדי למנוע מצב שבו טבלת מקור איקומרס מוצגת כלידים.

4. **בדיקות קבלה אחרי התיקון**
   - אין שגיאות TypeScript בבילד.
   - בטאב Facebook בדשבורד: הערכים והשורות תואמים לעמוד הטבלה של Facebook (בדיקת שורה-מול-שורה על תאריך/קמפיין/מזהה/חשיפות וכו’).
   - בחשבון מעורב (לידים + איקומרס): לא מתרחש ערבוב שגוי בין סוגי הקמפיינים.

### פרטים טכניים
- קבצים לשינוי:
  - `src/pages/SharedTable.tsx` (Build fix + הסרת תלות ב-`getCampaignType`)
  - `src/pages/DashboardView.tsx` (יישור תצוגת dashboard לתצוגת table מקורית)
- ללא שינויי DB / RLS / פונקציות Backend.
